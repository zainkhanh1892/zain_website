01 — Problem

Debugging AI NIC Clusters Is a Manual, Time-Intensive Process

AI NIC clusters generate enormous volumes of telemetry, logs, and diagnostic data. When something goes wrong, engineers manually search across logs, GitHub issues, internal documentation, and Jira tickets to find root causes — a process that routinely takes hours per incident.

At AMD's Networking Technology & Solutions Group, the knowledge needed to debug these clusters was scattered across disconnected systems. Engineers were spending most of their time on mechanical lookup tasks rather than the higher-order reasoning that actually requires human expertise.

02 — Goals

What I Set Out to Build

The goal was an agentic AI system that could reason over AMD's heterogeneous knowledge sources — telemetry, logs, internal docs, GitHub issues, and Jira tickets — and surface targeted root-cause candidates with traceable reasoning engineers could inspect and trust.

This meant solving three distinct problems: ingesting multimodal knowledge at scale, orchestrating tool-using agents that validate their own reasoning, and delivering usable UI prototypes for the team.

ComponentPurpose
RAG PipelineRetrieve the most relevant context from logs, telemetry, and docs at query time — giving the agent grounded signal instead of hallucinated answers.
Supervised Fine-Tuning (SFT)Adapt the base model to AMD-specific debugging patterns and terminology that general-purpose models don't know.
ReAct Agent FrameworkReason-then-act loops let the agent validate intermediate conclusions before committing to a root-cause hypothesis, catching errors before they surface to the engineer.
VectorDB Manager (Qdrant)Unified multimodal ingestion layer for docs, GitHub issues, and Jira tickets — scaled to 2,000+ issues while keeping retrieval latency stable.
03 — Process

How I Built It

I started by mapping AMD's existing debugging workflow end-to-end — understanding which knowledge sources engineers consulted, in what order, and where the slowest lookup steps were. That defined exactly what the retrieval system needed to cover before writing any code.

The pipeline ingests from multiple source types through a unified VectorDB Manager, embeds and indexes into Qdrant, then serves queries through ReAct agents that alternate between retrieval, reasoning, and validation steps before delivering a final hypothesis.

05 — Takeaways

What I Learned

  • 1RAG quality depends on chunking strategy as much as retrieval. How you split and embed documents has an outsized effect on whether the agent retrieves useful context. Chunk size and overlap experiments taught me that retrieval is a first-class engineering problem, not a configuration detail.
  • 2ReAct agents need validation loops to be trustworthy. Agents that skip self-validation produce confident but wrong outputs. Building reason-then-verify cycles before final answers was the key design decision that made the system usable in a real engineering context.
  • 3Domain-specific fine-tuning matters more than I expected. General-purpose models had blind spots around AMD-specific terminology and debugging patterns. SFT on internal examples meaningfully improved the precision of root-cause hypotheses.
  • 4Multimodal ingestion is harder than single-source retrieval. Unifying docs, GitHub, and Jira into one vector store required careful normalization — each source has different metadata, structure, and signal quality that affects retrieval relevance.
  • 5Engineers need to trust the reasoning, not just the answer. Delivering traceable, inspectable reasoning chains was as important as accuracy — an agent that gives the right answer for opaque reasons doesn't get adopted in a real workflow.
